# 工业级 WinForm 框架 —— 项目目录结构设计

> 本文档定义解决方案的整体分层、项目命名规范、文件夹组织方式，作为框架开发的基础约定。

---

## 一、命名规范

### 1.1 解决方案命名

假设你的公司/产品代号为 `Acme.Industrial`（请按实际替换），整体命名遵循：

```
{公司}.{产品线}.{模块}.{子模块}
```

例如：

- `Acme.Industrial.Core`
- `Acme.Industrial.Communication.Modbus`
- `Acme.Industrial.UI.Controls`

### 1.2 项目类型后缀约定

| 后缀 | 含义 | 示例 |
|------|------|------|
| `.Core` | 核心抽象、接口、基础模型 | `Acme.Industrial.Core` |
| `.Common` | 公共工具类、扩展方法 | `Acme.Industrial.Common` |
| `.Abstractions` | 仅接口定义 | `Acme.Industrial.Communication.Abstractions` |
| `.Infrastructure` | 基础设施实现 | `Acme.Industrial.Infrastructure` |
| `.UI` | 通用 UI 框架 | `Acme.Industrial.UI` |
| `.Controls` | 自定义控件库 | `Acme.Industrial.UI.Controls` |
| `.Modules.{XXX}` | 业务模块 | `Acme.Industrial.Modules.DeviceManagement` |
| `.Plugins.{XXX}` | 插件 | `Acme.Industrial.Plugins.ReportExporter` |
| `.Host` | 启动宿主项目 | `Acme.Industrial.Host` |
| `.Tests` | 单元测试 | `Acme.Industrial.Core.Tests` |
| `.IntegrationTests` | 集成测试 | `Acme.Industrial.Communication.IntegrationTests` |
| `.Tools.{XXX}` | 辅助工具 | `Acme.Industrial.Tools.ProtocolSimulator` |
| `.Samples.{XXX}` | 示例工程 | `Acme.Industrial.Samples.ModbusDemo` |
| `.Hmi` | HMI 组态系统 | `Acme.Industrial.UI.Hmi` |
| `.Scripting` | 脚本引擎 | `Acme.Industrial.Scripting` |

## 五、依赖关系约束

为避免循环依赖，定义清晰的依赖方向（**箭头方向 = 引用方向**）：

```
Host  ──►  Modules  ──►  Services  ──►  Communication  ──►  Infrastructure  ──►  Core/Common
                  │                                                    ▲
                  ├────────►  Scripting  ───────────────────────────────┤
                  │                                                    │
                  └────────►  UI.Hmi  ──►  UI.Controls  ───────────────┘
```

**原则**：

1. `Core` 和 `Common` 不能依赖任何业务项目
2. `Abstractions` 项目只能依赖 `Core`
3. `Modules` 之间**禁止直接依赖**，必须通过事件总线或服务接口通讯
4. `UI.Controls` 不能依赖 `Modules` 或 `Services`
5. `Scripting` 可以依赖 `Communication.Abstractions`（点位访问）
6. `UI.Hmi` 可以依赖 `Scripting.Abstractions`（脚本绑定）
7. 高层项目可以依赖低层，反之不可

## 十一、小结

本目录结构遵循以下原则：

- **分层清晰**：编号文件夹（01-Core ~ 12-Tests）保证视觉顺序
- **职责单一**：每个项目只做一件事
- **可测试**：每层都有对应测试项目
- **可扩展**：业务模块、协议、插件、脚本、HMI 均可独立增减
- **可维护**：命名规范、依赖约束防止架构腐化

下一步建议：参考《核心接口骨架》文档实现 `Core` 与 `Abstractions` 项目，再按《通讯协议层抽象设计》搭建协议层。
